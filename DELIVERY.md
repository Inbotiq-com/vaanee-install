# Vaanee On-Premise — Delivery Checklist

This is the operator runbook for shipping a Vaanee on-prem instance to a customer.
It captures every step that is **not** automated by `install.sh`, in order.

The product (6 repos) is built and enforced as follows:

| Concern | Where it lives | How it works |
|---|---|---|
| Customer app images | `vaanee-backend`, `vaanee-webhook`, `vaanee-frontend` (ACR) | Built from the `vaanee-onpremise` branches by `build-images.sh`, pulled by the installer |
| Central control | `inbotiq-backend` (qa→prod) `POST /api/vaanee/check-in` + `/usage` | Licence status, minute quota, concurrency, telephony + provider keys |
| Licence admin | `inbotiq-admin-backend` / `inbotiq-admin-frontend` (main) | Create licence, set limits, enter per-org keys |
| Local data | Customer's own PostgreSQL (pgvector) | Created by the installer migrations; KB embeddings stay local |

---

## 0. One-time central-DB change (do this ONCE, before any delivery)

Apply the canonical central-DB script in pgAdmin against the **central** database
(`inbotiq_qa_db` — the one the admin portal + `/api/vaanee/check-in` use):

```
docs/VAANEE_central_db_pgadmin.sql
```

It is idempotent. It adds the 3 per-org provider-key columns
(`cartesia_api_key`, `sarvam_api_key`, `google_api_key`) and refreshes the
licence-limit sync function. **Until this runs, the check-in delivers NO telephony**
(the telephony query selects those columns and fails closed to `telephony=null`),
so no customer VM can place calls.

## 1. Rotate the leaked secrets (before go-live)

These were committed to source historically and MUST be rotated before delivery:

- **Provider keys** in `AI_Webhook/.github/workflows/deploy-qa.yml` (Cartesia, Azure
  Speech). These are qa CI app-settings, not in the on-prem image, but rotate them.
- **ACR pull token** (was hardcoded in `lib/config.sh`; now removed). Generate a
  **scoped, per-customer** ACR token (see step 3).
- Any provider keys you previously tested with in the central DB.

The on-prem **images contain no secrets** — per-install secrets (`JWT_SECRET`,
`ENCRYPTION_*`) are generated by the installer into `~/vaanee/.env`; per-org provider
keys arrive (decrypted, over TLS) via check-in.

## 2. Build + publish the image set

On a machine with `az login` (push rights) and the three repos checked out on
`vaanee-onpremise`:

```bash
ONPREM_IMAGE_TAG=YYYY-MM-DD-onprem ./build-images.sh /path/to/src-root
```

Builds `vaanee-webhook`, `vaanee-backend`, `vaanee-frontend` cloud-side (`az acr
build`) under ONE tag. Then set the same tag in `lib/config.sh`:

```bash
ONPREM_IMAGE_TAG="YYYY-MM-DD-onprem"
```

## 3. Provision the customer licence + ACR token

- In the **admin portal**: create the customer org + Vaanee licence; set plan
  limits (minute allotment, max agents, max concurrent calls). The licence's
  `vaan_live_...` API key goes into the VM `.env` as `VAANEE_API_KEY`.
- Generate a **scoped ACR pull token** for the customer (so each customer has its
  own revocable credential), to be passed at install time.

## 4. Enter per-org provider + telephony keys (admin portal → org page)

On the org's telephony page enter the org's own:
- **Exotel** (account SID, API key, API token, app/flow id) **or Plivo** (auth id,
  auth token) — pick the provider; bind the org's phone number(s).
- **Assigned Keys**: Cartesia, Sarvam, Google (masked inputs). These are encrypted
  centrally and delivered to the VM via check-in; the VM uses them directly.

## 5. Install on the customer VM

Requirements: Ubuntu, Docker + Compose, a reachable PostgreSQL with **pgvector**
available (the installer preflights this), DNS for the customer domain.

```bash
REGISTRY_PASS='<per-customer ACR token>' \
VAANEE_API_KEY='vaan_live_...' \
VAANEE_DOMAIN='customer.example.com' \
DATABASE_URL='postgres://...' \
ADMIN_EMAIL='ops@inbotiq.com' \
bash install.sh
```

The installer: writes `~/vaanee/{.env,docker-compose.yml,Caddyfile}`, runs the
PostgreSQL migrations (001–008, pgvector preflight first), pulls the pinned image
set, and brings the stack up behind Caddy (Let's Encrypt).

## 6. Verify + hand off

- `vaanee status` → all containers healthy.
- `vaanee logs checkin` → "Check-in successful — status: active" and a telephony
  line with the right provider.
- In the portal, confirm the licence shows `last_checkin_at` updating.
- **Call tests** (manual): place an outbound call and receive an inbound call;
  for inbound, confirm the bot transfers to the saved number.

---

### Enforcement notes (how central control holds)

- **Kill switch**: pausing / cancelling / expiring the licence in the portal stops
  the VM from placing calls within one check-in interval (the call path reads the
  cached status and fails closed).
- **Quota**: per-minute usage is reported each call; calls stop when the minute
  allotment is exhausted.
- **Anti-detachment**: a VM cut off from the central server keeps serving only for
  a bounded grace window (`VAANEE_LICENCE_MAX_STALE_SECONDS`), then fails closed.
